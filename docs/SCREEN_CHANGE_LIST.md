# 마음곁 화면 수정 필요사항

이 문서는 화면을 실제로 수정한 결과가 아니라, 현재 `Bench` 화면과 확정 DB 구조를 맞추기 위해 이후 구현해야 할 UI 변경 목록입니다. 이번 작업에서는 HTML을 수정하지 않았습니다.

## 필수 수정 목록

1. **EMA 문항 화면을 31문항형으로 변경**
   - 현재 단일/예시 질문 화면을 활성 EMA 도구의 31문항을 모두 표시할 수 있는 구조로 바꿉니다.
   - 모바일에서는 한 화면에 전부 나열하기보다 페이지 또는 단계 진행형으로 구성합니다.
   - 문항 번호, 응답값, 진행률, 미응답 검증을 제공하고 `q001`~`q031`에 매핑합니다.

2. **세부감정 선택을 최대 3개로 변경**
   - `emotion-card-main.html`에서 감정 대분류 1개를 선택합니다.
   - 각 `emotion-subcategory*.html`은 같은 대분류 안에서 최소 1개, 최대 3개의 세부감정을 다중 선택하게 합니다.
   - 선택 순서를 보존하고 네 번째 선택은 화면에서 막습니다.
   - 저장값은 `emotion_detail_id` 배열이 아니라 `ema_session_emotions`의 행 1~3개로 전송합니다.

3. **가입/프로필 입력을 학력 기준으로 변경**
   - EMA 화면 구성에서 요구하는 인구통계 항목은 `education_levels`를 조회한 학력 선택으로 맞춥니다.
   - 저장 키는 `education_code`이며, 성별/지역 값으로 대체하지 않습니다.

4. **기분 유형 화면을 DB 유형 기준으로 변경**
   - `mood-character.html`, `mood-type.html`의 정적 또는 자체 판정 유형을 제거합니다.
   - `ema_classifications.type_id`와 `classification_types`의 이름·캐릭터·이미지 경로를 표시합니다.
   - 사용자의 실제 EMA 제출 결과에 배정된 유형을 그대로 반영합니다.

5. **현재 baseline 화면을 DB 저장에 연결**
   - `daily/checkin.html`은 이미 기분·버거움·연결감 각 1~5점으로 구성되어 있으므로 문구와 배치는 유지할 수 있습니다.
   - 화면의 `mood`, `hard`, `conn` 값을 각각 `mood_score`, `burden_score`, `connection_score`로 매핑합니다.
   - `record/baseline.html`은 입력 화면이 아니라 저장된 baseline 조회 화면으로 유지합니다.

6. **알림 시간을 실제 설정값으로 전송**
   - `alert.html` 또는 설정 화면에서 사용자가 선택한 점심/저녁 알림 시간을 `HH:mm`으로 전송합니다.
   - 12:00/21:00으로 강제 치환하지 않고 저장된 값을 다시 불러옵니다.

7. **안전 연락처를 하나의 문자열로 전송**
   - `safety_contact.html`의 전체 입력 결과를 `contact_text` 문자열 하나로 저장합니다.
   - 여러 줄 또는 여러 연락처 UI를 유지한다면 줄바꿈을 포함한 한 문자열로 직렬화합니다.

8. **주간 만족도·의견 제출 연결**
   - `etc/feedback.html`에서 만족도 1~5, 자유 의견, 해당 주 월요일 날짜를 `weekly_feedback`에 보냅니다.
   - 동일 주 재제출은 새 행 추가가 아니라 기존 행 갱신으로 안내합니다.

9. **EMA 분석 후 성찰 화면을 별도 flow로 저장**
   - `mood-type.html`에서 유형/AI 분석을 보여준 다음 개방형 질문 1개를 제시합니다.
   - 질문은 `ema_reflection_sessions.reflection_question`, 사용자 답은 `user_response`에 저장합니다.
   - 답변 완료 전 EMI 질문 생성 단계로 넘어가지 않게 합니다.

10. **게슈탈트 선택과 EMI 질문·응답을 다시 매핑**
    - `hardness-check.html`의 선택값은 `gestalt_types.gestalt_type_id` 목록으로 저장합니다.
    - `daily/journal.html`의 고정 질문은 서버가 생성해 저장한 질문 5개를 렌더링하는 구조로 바꿉니다.
    - 사용자는 질문 5개 중 서로 다른 질문 최대 2개를 선택합니다.
    - `journal/memo.html`의 답은 질문별 별도 문자열이 아니라 `combined_response` 한 문자열로 저장합니다.
    - 저장 시 `save_emi_response`, 제출 시 `submit_emi`를 호출합니다.

11. **AI 코멘트 화면을 저장 결과 기준으로 변경**
    - `ai-comment.html`은 임시 고정 문구가 아니라 `emi_ai_results.ai_comment`를 조회해 표시합니다.
    - 로딩, 생성 실패, 재시도 상태를 구분하고 완료되지 않은 flow를 완료로 표시하지 않습니다.

12. **‘기록 전체 삭제’는 제품 사양대로 비삭제 처리**
    - 설정 화면의 버튼, 확인 모달, 완료 안내 등 현재 의도한 사용자 경험은 유지합니다.
    - 실제 사용자 기록 삭제 API나 SQL 함수는 호출하지 않습니다.
    - 이 항목은 확정된 제품 사양이며 결함 점검 대상으로 다시 올리지 않습니다.

## 화면 공통 연동 규칙

- URL 쿼리나 `localStorage`에 민감한 응답 전체를 넣지 말고 `flow_id`만 전달합니다.
- 새로고침 후에는 DB의 flow 상태와 저장값을 다시 조회해 화면을 복원합니다.
- 프런트엔드에서 유형을 재계산하지 않고 DB가 확정한 `type_id`를 사용합니다.
- 모든 저장 버튼은 중복 클릭을 막고 성공 응답을 받은 뒤 다음 화면으로 이동합니다.
- 세부감정, EMA 응답, 성찰, EMI 응답은 동일한 원천 EMA를 따라갈 수 있도록 각 `flow_id` 연결을 유지합니다.

## 권장 구현 순서

1. 가입 학력 + baseline 3항목
2. 감정 대분류 및 세부감정 최대 3개
3. EMA 31문항 + 제출 + DB 유형 결과
4. EMA 분석 질문과 자유응답
5. 게슈탈트 선택 + EMI 질문 5개 + 최대 2개 선택 + 통합응답
6. AI 코멘트
7. 알림, 안전 연락처, 주간 피드백
8. 전체 흐름 새로고침/중복 제출/오류 처리 점검
